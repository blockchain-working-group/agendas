Inter-Agency Adoption Framework

A repeatable playbook for standing up blockchain pilots between two federal agencies. Derived from JFMIP-24-01 operational considerations and reframed for speed, demonstrated value, and reuse.

Design Principles

  • 1. Sequence over completeness. Distinguish what's needed before a pilot vs. before production vs. before the third agency joins.
  • 2. Organizational agreements are the bottleneck. Two CISOs agreeing on shared risk is harder than deploying a node. Make agreements lightweight, modular, reusable.
  • 3. Demonstrated value creates momentum. Front-load visible output. Every week without a working demo is a week leadership attention drifts.
  • 4. Templates, not bespoke documents. Every artifact from the first pairing becomes a template for the next.
0

Pre-Commitment

1–2 weeks

Two agencies agree to pilot. Not to build a production system. Not to commit budget. To run a time-boxed experiment and publish what they learn.

Identify champions

Name a point of contact at each agency from the program office, not IT. These are the people who own the use case and can make decisions without committee approval.

Scope the use case

Agree on a use case narrow enough to pilot in 30 days. Produce a one-page brief: what data moves, between whom, why it matters, and what success looks like.

Sandbox classification

Both CISOs acknowledge in writing that the pilot operates on synthetic data, outside production boundaries, and does not require a full ATO. This is the single highest-leverage decision in the framework — it unblocks everything else.

Sign a lightweight ISA

A 2–3 page information sharing agreement covering data types, permitted uses, duration, and points of contact. Adapted from the Fiscal Service / GAO ISA, stripped to essentials. Becomes a template for every subsequent pairing.

Definition of Done — Phase 0

  • ☐ Named champion at each agency with decision authority for the pilot
  • ☐ One-page use case brief signed by both champions
  • ☐ Sandbox classification memo signed by both CISOs confirming synthetic data, non-production, time-boxed scope, no full ATO required
  • ☐ Lightweight ISA executed by both agencies
  • ☐ All three documents committed to a shared repository
1

First Transaction

2–4 weeks

Both agencies complete one full cycle of the use case on a shared ledger. Humans in the loop for every step. No automation.

JFMIP Category What We Do Now What We Defer
Business Processes Map one use case flow. Define roles: who initiates, approves, verifies. Process automation, system integration
Data Standardization Minimal schema for this use case only. Versioned JSON schema. Government-wide naming conventions, metadata standards
Testing Manual walkthrough. Document every friction point. Load testing, automated suites
User Experience Both participants use the same tool. Collect qualitative feedback. Polished UI, training materials
Security Keypair per participant. Application-layer encryption. No classified data. HSM, key rotation, MFA
Infrastructure Public chain + SPV verification. No agency-hosted nodes. Redundancy, disaster recovery
Interoperability Self-contained pilot. Document which systems would need interfaces. API connections to production systems
Best Practices Categorize as low-impact (synthetic data, no real money, time-boxed). Full NIST RMF, continuous monitoring

Definition of Done — Phase 1

  • ☐ At least one complete use case cycle recorded on-chain (e.g., award → accept → drawdown → disburse → report)
  • ☐ Both agencies independently verified the transaction chain via SPV / Merkle proofs
  • ☐ Neither participant can unilaterally alter the record of any completed step
  • ☐ Versioned JSON data schema published in the shared repository
  • ☐ Friction log completed: every organizational, technical, and procedural obstacle documented in sequence
  • ☐ Cost ledger completed: transaction fees, development hours, meeting hours, agreement negotiation hours
2

Governance & Audit

2–4 weeks, overlapping Phase 1

Establish the lightest viable governance structure and demonstrate auditor access. The JFMIP recommended a layered/mixed model with committees and an executive board — that's the right model at scale. For two agencies, it's overkill.

Two-Agency Governance

  • Decisions: Mutual consent, 48-hour default-to-yes
  • Disputes: Escalate to each agency's champion
  • Schema changes: Pull request to shared repo, both parties review
  • Funding: Each agency covers own costs (sub-$100 in fees; staff time is the real cost)

Auditor Access

  • Stand up a read-only verification view that reconstructs the full transaction chain for any award ID
  • Invite a third participant (auditor or IG rep) to use it against Phase 1 data
  • Document what the auditor can verify, what they can't, and what additional evidence they'd need

Definition of Done — Phase 2

  • ☐ Two-page governance agreement executed: decision authority, dispute resolution, schema change process, cost allocation
  • ☐ Read-only auditor view functional — takes an award ID, returns the full verified transaction chain
  • ☐ At least one person outside the two pilot agencies has used the auditor view and provided feedback
  • ☐ Auditor access guide published: what SPV proves, what it doesn't, what supplementary evidence is needed
  • ☐ Onboarding playbook drafted: every step from "two agencies want to try this" to "first verified transaction"
3

Retrospective & Publish

1 week

Turn the pilot into a reusable asset. The artifacts produced here are the product — they're what makes the next pairing faster.

Joint retrospective

Both agencies meet: what worked, what didn't, what took longer than expected, what was easier than expected. Raw and honest, not polished.

Publish everything

Friction log, ISA template, governance agreement, data schema, auditor guide, onboarding playbook — all in a public repository.

Quantify the pilot

Number of transactions, total on-chain cost, elapsed calendar time, person-hours, number of meetings, number of documents signed.

Identify the next pairing

Which agency joins next? What use case? What in the templates needs to change for their context?

Definition of Done — Phase 3

  • ☐ Joint retrospective completed with both agencies present
  • ☐ Friction log published — raw sequence of organizational, technical, and procedural obstacles
  • ☐ All template artifacts (ISA, governance, schema, auditor guide, onboarding playbook) published to public repository
  • ☐ Pilot quantified: transactions, cost, time, person-hours, meetings, documents
  • ☐ Next pairing identified: agency, use case, and which templates need adaptation
4

Repeat

Target: half the elapsed time of the previous pairing

Agency C joins. The framework should make it faster. The key metric is time-to-first-transaction for a new agency joining the network.

Artifact First Pairing Second Pairing Third+
ISA Negotiated from scratch Adapt template (swap names, review data types) Template stabilizes; signing is administrative
Data Schema Invented for the use case Extended (new fields, backward-compatible) Versioned governance formalizes
Verification Tooling Built from scratch New agency uses existing auditor view CI/CD for schema validation
Governance Mutual consent (2 parties) Lightweight charter (3 parties need a tiebreaker) Layered/mixed model activates
Sandbox Memo Written from scratch Precedent exists; new CISO reviews prior memo Sandbox → pilot → production path documented
Friction Log Full log Shorter (known issues flagged in advance) Becomes a changelog

Definition of Done — Phase 4

  • ☐ Agency C completed a full use case cycle using the existing templates and tooling
  • ☐ Time-to-first-transaction for Agency C was measurably shorter than the first pairing
  • ☐ Templates updated with changes needed for the new pairing and committed back to the repository
  • ☐ Governance model expanded to handle three participants (tiebreaker mechanism in place)
  • ☐ Onboarding playbook validated: Agency C followed it and it worked (or the gaps are documented)
  • ☐ Decision recorded: continue adding agencies, or transition the sandbox toward a production pilot with real data / real money

Source

This framework is derived from the operational considerations in JFMIP-24-01 (pp. 45–58), resequenced around the principle that organizational agreements — not technology — are the bottleneck, and that the fastest way to reduce friction is to do the work once, publish everything, and compress each repetition.
